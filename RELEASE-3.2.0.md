## Что изменилось

### Заимствование объектов в расширение — больше не ломает проект

**Было:** при заимствовании объекта (документа, справочника, регистра) из основной конфигурации в расширение через AI-агента — операция формально проходила успешно, объект появлялся в дереве расширения, никаких визуальных ошибок не было. Но при попытке обновить информационную базу всё падало с ошибкой:

```text
Файл объекта не существует - /Catalogs/Партнеры.xml
Файл объекта не существует - /DefinedTypes/Артикул.xml
... и так далее
```

Расширение становилось неработоспособным, и без ручного вмешательства починить его было нельзя.

**Стало:** заимствование объектов в расширение теперь работает корректно — обновление информационной базы проходит без ошибок «Файл объекта не существует».

---

### Самовосстановление сломанных расширений

Это самое важное для тех, у кого уже есть расширения, испорченные предыдущей версией плагина.

**После обновления плагина любое сломанное расширение можно починить одним запросом AI-агенту.** Вмешиваться в файлы или повторять заимствование вручную не нужно — AI-агент сам разберётся и за один вызов восстановит расширение.

---

### Красивые заголовки полей формы

**Было:** когда AI-агент добавлял реквизит на форму, заголовок поля получал значение имени реквизита — без пробелов и слитно. Например, если у реквизита было имя `ДополнительнаяИнформацияВодителя` и синоним «Дополнительная информация для водителя», на форме отображался **некрасивый** заголовок `ДополнительнаяИнформацияВодителя`. Чтобы получить нормальный заголовок, приходилось править вручную.

**Стало:** заголовок поля автоматически берётся из синонима связанного реквизита — точно так же, как при добавлении поля кнопкой «+» в дереве формы EDT. Указанный пример теперь сразу отобразится как «Дополнительная информация для водителя». Если нужен какой-то другой заголовок, отличный от синонима, его можно явно задать через AI-агента.

---

### Константы с любым типом данных

**Было:** когда AI-агент создавал константу, тип данных всегда был «Строка длиной 10», даже если запрашивался другой. Игнорировались все попытки задать число, булево, дату или ссылку на справочник.

**Стало:** константу теперь можно создать с любым типом данных:

- Примитивные: Строка, Число, Булево, Дата
- Ссылочные: ссылка на справочник, документ или перечисление основной конфигурации

Если для константы указан ссылочный тип, и сам объект (например, справочник) ещё не заимствован в расширение, он будет автоматически заимствован вместе с константой — никаких дополнительных действий не требуется. Например, можно сказать AI-агенту:

> «Добавь в моё расширение константу `СтранаПоУмолчанию` со ссылкой на справочник стран мира»

— и всё сработает за один шаг, даже если справочника ещё не было в расширении.

## Как починить уже сломанное расширение

Если у тебя есть расширение, в котором обновление информационной базы падает с ошибкой «Файл объекта не существует»:

1. **Установи новую версию плагина 3.2.0.**
2. **Открой проект расширения в EDT.**
3. **Скажи AI-агенту в чате:**

   > «У моего расширения `<имя_твоего_расширения>` обновление информационной базы падает с ошибкой "Файл объекта не существует". Почини, пожалуйста.»

4. **Запусти обновление информационной базы заново.** Должно пройти без ошибок.

Один запрос чинит расширение целиком — независимо от того, сколько объектов в нём было повреждено.

---

## Подводный камень: пересоздание расширения с тем же именем

Если ты удалишь расширение из EDT и **создашь новое расширение с тем же именем** — EDT может сохранить часть кэша от старого расширения. Внешне всё выглядит правильно, обновление базы проходит — но в режиме предприятия новые поля формы могут не отображаться.

**Решение:** при пересоздании расширения дай ему **новое имя** (например, `МоёРасширение` → `МоёРасширение2`).

Это особенность Eclipse и EDT, не плагина. Просто полезно знать.
