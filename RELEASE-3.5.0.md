## Расширения: код в модулях заимствованных объектов

### Автоматическое включение модуля при записи кода

**Как было.** Попросил агента в расширении дописать `&После("ОбработкаПроведения")` на заимствованный документ — он вызывал `write_module_source` на `ObjectModule`, получал `success: true`, валидация зелёная, файл `ObjectModule.bsl` на диске. В EDT нажимаешь «Обновить конфигурацию базы данных расширения» — проходит без ошибок. Открываешь документ в пользовательском режиме, проводишь — и код не срабатывает. Ни `Сообщить()` не выполняется, ни движения не формируются, отладчик на точку останова не встаёт.

**Причина.** В расширении у заимствованного объекта каждый модуль (объектный, менеджера, набора записей, команды, менеджера значения) должен быть явно помечен как «участвующий в расширении» — в редакторе объекта на вкладке «Расширение» есть раздел «Свойства, участвующие в расширении» с отдельной галочкой на каждый модуль. Без этой галочки `.bsl`-файл на диске есть, валидация его не ругает, но платформа при исполнении в 1С:Предприятии такой модуль игнорирует. Когда заимствование выполнял сам пользователь через EDT, он обычно ставил галочку руками. Когда заимствование делал плагин через `adoptObject` / `adoptObjects` / `adoptChild` — галочка не выставлялась. Результат — «тихий баг»: код написан и сохранён, но в предприятии не работает.

**Что стало.** Плагин сам включает модуль в расширении при записи в него кода. После успешного `write_module_source` на модуль заимствованного объекта расширения выставляется ровно та галочка, которая соответствует типу модуля (Объектный / Менеджера / Набора записей / Команды / Менеджера значения / модуль общего модуля). Отдельно в EDT или через MCP это делать не нужно.

Результат активации возвращается в ответе `write_module_source` в поле `moduleActivation`:

```json
"moduleActivation": {
  "moduleType": "ObjectModule",
  "baseFeatureName": "objectModule",
  "belonging": "ADOPTED",
  "previousState": "NONE",
  "activated": true,
  "newState": "EXTENDED"
}
```

Для ситуаций, где активация не нужна (модуль основной конфигурации, модуль native-объекта расширения, модуль формы — он управляется иначе), в ответе будет `activated: "notApplicable"` с пояснением — никаких изменений, никаких ошибок. Операция идемпотентна: повторная запись в уже активный модуль вернёт `alreadyActivated: true`.

Типичный сценарий теперь выглядит так:

```json
// 1. Заимствовать документ в расширение
{"operation":"adoptObject","projectName":"МоёРасширение",
 "objectName":"Document.ЗаказКлиента"}

// 2. Записать код в его объектный модуль — активация произойдёт сама
// write_module_source modulePath=Documents/ЗаказКлиента/ObjectModule.bsl ...
// В ответе: moduleActivation.activated = true, newState = EXTENDED
```

После этого «Обновить конфигурацию базы данных расширения» в EDT и перезапуск предприятия — код работает.

### Новая операция `adoptModule` для включения без написания кода

**Как было.** Если агент хотел включить модуль заимствованного объекта, не добавляя своего кода (например, чтобы унаследовать поведение базового модуля as-is), единственный путь был — открыть объект расширения в EDT, перейти на вкладку «Расширение» и поставить галочку руками. Через MCP это не автоматизировалось.

**Что стало.** Новая операция `edit_metadata operation=adoptModule` включает нужную галочку программно. Параметры — `projectName` (расширение), `objectName` (FQN заимствованного объекта) и `moduleType` (какой именно модуль). Операция работает для любых объектов и типов модулей, которые платформа разрешает расширять: ObjectModule, ManagerModule, RecordSetModule, CommandModule, ValueManagerModule (у констант), Module (у общих модулей).

Пример — включить модуль набора записей у заимствованного регистра накопления:

```json
{"operation":"adoptModule","projectName":"МоёРасширение",
 "objectName":"AccumulationRegister.ТоварыНаСкладах",
 "moduleType":"RecordSetModule"}
```

Операция идемпотентна: повторный вызов с тем же `moduleType` вернёт `alreadyActivated: true` и `success: true`, никаких изменений. Для объектов основной конфигурации, native-объектов расширения или не поддерживаемого для данного типа модуля (например, `RecordSetModule` для документа) — `activated: "notApplicable"` с пояснением.

Типичный случай, когда эта операция нужна — только редкий сценарий «унаследовать базовое поведение». В обычной работе при записи кода через `write_module_source` активация происходит автоматически, и отдельный вызов `adoptModule` не требуется.

## Подписки на события

### Создание подписки одним вызовом

**Что это и зачем.** Раньше операция `createObject` на подписке на событие сразу отказывала — мол, создание `EventSubscription` не поддерживается, создавайте руками в EDT. А это ключевой механизм «подпишись на проведение документа и выполняй своё», без него агент, которому дали задачу «при проведении заказов делать X», не мог довести до конца. Теперь подписка создаётся одним вызовом `createObject` — с источником, именем события и обработчиком.

**Как пользоваться.** Минимальный пример:

```json
{"operation":"createObject","projectName":"МойПроект",
 "objectName":"EventSubscription.ПроведениеЗаказа",
 "eventSource":"DocumentObject.Заказ",
 "eventName":"ОбработкаПроведения",
 "handler":"ОбщийМодульПодписок.ПриПроведенииЗаказа"}
```

Имя события принимается как по-русски («ОбработкаПроведения»), так и по-английски («Posting»). В `.mdo` сохраняется каноническое английское имя — в ответе плагина поле `eventName` возвращает его, чтобы агент увидел, к чему пришло значение.

Развёрнутые сценарии (массив источников, расширение, обработка ошибок) — `help topic=createObjectEventSubscription`.

### Массив источников в одной подписке

**Что это и зачем.** Типовая задача — «подписать три документа на одно событие с одним обработчиком». Раньше это не делалось никак. Теперь параметр `eventSource` принимает и строку, и массив FQN — тогда подписка ловит событие у нескольких источников одновременно, одним обработчиком. Плагин заранее проверяет, что выбранное событие есть у всех указанных источников, и если хоть у одного нет — отказывает со списком общих событий.

**Как пользоваться.** Одна подписка на три документа:

```json
{"operation":"createObject","projectName":"МойПроект",
 "objectName":"EventSubscription.ДвиженияПоЗаказам",
 "eventSource":[
   "DocumentObject.Заказ",
   "DocumentObject.ПоступлениеТоваров",
   "DocumentObject.ПриходныйКассовыйОрдер"
 ],
 "eventName":"ОбработкаПроведения",
 "handler":"ОбщийМодульПодписок.ПриПроведенииЛюбогоИзТрёхДокументов"}
```

В ответе — поле `eventSourceCount: 3`. Для расширения источник из базовой конфигурации работает напрямую — заимствовать объект не нужно; если тип найден в другом проекте, в ответе появится `eventSourceResolvedFrom` с его именем.

### Заглушка-обработчик в общем модуле

**Что это и зачем.** После создания подписки в указанный общий модуль **автоматически** дописывается процедура-заглушка обработчика. С правильной сигнатурой — первым параметром идёт `Источник` (сам проводимый документ/объект), дальше параметры события (`Отказ`, `РежимПроведения` и т. п.), в конце — `Экспорт`. Агенту не нужно угадывать имена параметров: они берутся из описания события платформы, русские — если в конфигурации русские имена. Тело заглушки — комментарий `// TODO: реализуйте логику обработчика`, дальше агент просто пишет код через `write_module_source`.

Операция идемпотентна: если процедура с таким именем в модуле уже есть — она не трогается, в ответе `handlerAlreadyExists: true`. При `dryRun: true` заглушку не пишем — только сообщаем план.

**Пример сгенерированной заглушки.** Для `createObject EventSubscription` на `DocumentObject.Заказ` / событие `ОбработкаПроведения` / обработчик `ОбщийМодульПодписок.ПриПроведенииЗаказа`:

```bsl
// Обработчик события "ОбработкаПроведения" источника DocumentObject.Заказ
// Подписка: EventSubscription.ПроведениеЗаказа
Процедура ПриПроведенииЗаказа(Источник, Отказ, РежимПроведения) Экспорт

	// TODO: реализуйте логику обработчика

КонецПроцедуры
```

В ответе плагина — поля `handlerStubCreated`, `handlerModulePath`.

### Отдельная операция `addEventSubscriptionHandler`

**Что это и зачем.** Бывает, что подписка уже создана, но общего модуля на момент создания ещё не было (или изменилось имя метода в `handler`). Тогда заглушки в модуле нет, и агент должен её написать руками с правильной сигнатурой — а при «слабом» агенте это лотерея. Новая операция `addEventSubscriptionHandler` дописывает заглушку по уже существующей подписке: плагин сам читает из неё источник, событие и обработчик, считает сигнатуру и пишет процедуру в общий модуль.

**Как пользоваться.** Минимальный вызов:

```json
{"operation":"addEventSubscriptionHandler","projectName":"МойПроект",
 "objectName":"EventSubscription.ПроведениеЗаказа"}
```

В ответе — `stubStatus: "CREATED"` и `handlerModulePath` — путь к изменённому модулю. Если процедура с таким именем уже была — `stubStatus: "ALREADY_EXISTS"` (не трогаем). Параметр `force: true` перезаписывает существующую процедуру свежей заглушкой (старый блок `Процедура…КонецПроцедуры` удаляется). `dryRun: true` — запись не идёт, но в ответе возвращается `handlerPlannedSignature` с планируемой строкой `Процедура X(…) Экспорт` — удобно проверить сигнатуру до записи.

## Создание общих модулей

### Контексты исполнения (Сервер / Клиент / Вызов сервера)

**Что это и зачем.** Раньше `createObject CommonModule` создавал модуль с одним включённым флагом «Сервер» и всё. Остальные контексты (Клиент управляемого приложения, Вызов сервера, Привилегированный и т. п.) приходилось доставать вручную в EDT — а это снова выбивало агента из рабочего цикла. Теперь у `createObject CommonModule` есть отдельные параметры на каждую галочку — агент явно указывает, что за модуль ему нужен, и плагин проставляет галочки правильно и полностью. Дополнительно теперь все семь флагов (Сервер / Клиент упр. / Клиент обычн. / Внешнее соединение / Вызов сервера / Привилегированный / Глобальный) записываются в файл объекта явно, без «полупустого» состояния — раньше EDT иногда подсвечивал свеже-созданный модуль жёлтым треугольником «некорректные настройки», теперь не подсвечивает.

**Как пользоваться.** Самый типичный случай — серверный модуль с разрешённым вызовом с клиента:

```json
{"operation":"createObject","projectName":"МойПроект",
 "objectName":"CommonModule.ВызовСервера",
 "server":true,"serverCall":true}
```

Клиент-серверный модуль (доступен и на клиенте, и на сервере, без вызова экспортных методов с клиента):

```json
{"operation":"createObject","projectName":"МойПроект",
 "objectName":"CommonModule.ОбщийКлиентСервер",
 "server":true,"clientManagedApplication":true}
```

Параметр `returnValuesReuse` управляет «Повторным использованием возвращаемых значений» — допустимы `DontUse` (дефолт), `DuringRequest`, `DuringSession`, либо русские «НеИспользовать» / «НаВремяВызова» / «НаВремяСеанса». В ответе плагина поле `commonModuleFlags` показывает применённые значения — это те флаги, которые агент передал явно (дефолты в ответ не выводятся).

## Защита от дублей при добавлении реквизитов, ТЧ, полей регистра и значений перечисления

### Повторный вызов не создаёт одноимённый элемент

**Как было.** Пользователь запускает агента сгенерировать реквизиты объекта, агента прерывает («Стоп»), через минуту говорит «продолжи». Агент начинает генерацию заново — и в `.mdo` появляется второй `Комментарий`, второй `ДатаЗаказа`, второй `Контрагент`. Или более частый случай — агент получает задачу «создай два документа с одинаковыми колонками ТЧ», `addTabularSectionAttribute` вызывается дважды, в результате в одной табличной части оказывается два реквизита с именем `Количество`. Платформа такое молча принимает, но работать с объектом невозможно — дубли приходится удалять руками через EDT.

**Причина.** Пять операций `edit_metadata` (`addObjectAttribute`, `addTabularSection`, `addTabularSectionAttribute`, `addEnumValue`, `addRegisterField`) не проверяли, нет ли уже элемента с таким именем в целевой коллекции, и просто добавляли новый. Для агента вызов идемпотентен, если плагин при повторе с теми же параметрами не плодит дубль — а проверки не было.

**Что стало.** Все пять операций теперь проверяют, есть ли элемент с таким же именем перед добавлением. Если есть — новый не создаётся, ответ приходит в едином формате:

- `success: true` — операция отработала (а не отказ: повторный вызов — это не ошибка, это idempotent).
- `alreadyExists: true` — пометка, что хотя бы одно имя совпало.
- `existing: "<kind>:<name>"` — в одиночном режиме, описание того, что уже было.
- `skipped: "имя1, имя2"` и `skippedCount: N` — в пакетном режиме, список пропущенных имён.

В смешанном батче (часть имён уже есть, часть — новые) существующие попадают в `skipped`, а новые создаются и попадают в `created` как обычно — агент получает полную картину, не нужно разбирать это отдельными вызовами.

Пример: запустили агента, он начал создавать три реквизита, `Комментарий` успел записаться, генерацию прервали. Запустили заново с теми же тремя именами — плагин вернёт `skipped: "Комментарий"` и `created: "СуммаЗаявки, Автор"`. Ничего не дублируется, ничего не теряется.

Для `addRegisterField` это уточнение — раньше операция при повторе возвращала `success: false` с текстом ошибки про дубликат, теперь формат ответа приведён к общему стилю. Проверка в пределах одного `kind`: ресурс `Сумма` и реквизит `Сумма` — разные поля (платформа хранит их отдельно), дубля при таких именах не будет.

## Таблицы на форме: уникальные имена колонок при нескольких ТЧ

### Автогенерация колонок через EDT-wizard

**Как было.** Агент просит расположить на форме две табличные части с одинаковыми названиями колонок (типовой случай — документ с ТЧ `Товары` и ТЧ `Услуги`, у обеих колонки `Наименование`, `Количество`). Делает два вызова `addTable` — один на `Объект.Товары`, второй на `Объект.Услуги`. Плагин оба раза отвечает `success: true`, EDT-валидатор на форму не ругается. Но на форме оказываются **два элемента** с именем `Наименование` и **два** с именем `Количество`. Запись `Элементы.Наименование.Видимость = Ложь` в модуле формы становится неоднозначной — платформа обращается к первому попавшемуся, и поведение непредсказуемо.

**Причина.** У `addTable` при автогенерации колонок (параметр `fields` не передан) плагин брал имена FormField напрямую из имён реквизитов табличной части. Два столбца `Наименование` в одном `.form`-файле получали буквально имя `Наименование` — коллизия. При этом когда агент создаёт форму через `createForm layout=auto` (мастер «Добавить форму» в EDT), у таких же колонок появляется префикс имени ТЧ: `ТоварыНаименование`, `УслугиНаименование` — и коллизии нет. Разные пути работы с формой давали разные имена.

**Что стало.** `addTable` при автогенерации колонок теперь использует тот же механизм, что и мастер EDT — нативный сервис генерации форм. Колонкам даются имена с префиксом ТЧ: `ТоварыНаименование` / `ТоварыКоличество` для ТЧ `Товары`, `УслугиНаименование` / `УслугиКоличество` для ТЧ `Услуги`. Две таблицы с одноимёнными колонками на одной форме теперь дают восемь разных элементов формы, `Элементы.ТоварыНаименование` и `Элементы.УслугиНаименование` однозначно адресуют нужное поле.

В коде модуля формы обращаться к колонкам таких таблиц теперь нужно по имени с префиксом: `Элементы.ТоварыНаименование.Видимость = Ложь` — это и раньше был правильный паттерн при создании форм через EDT, но `addTable` от него отклонялся. Если вы передаёте `fields=[...]` явно — имена берутся из массива как есть, ответственность за уникальность на агенте (префикс можно прописать в `fields[i].name`).

Для типа `addDynamicListTable` (динамический список) изменение не применяется — там другой API-путь генерации колонок. Случай «две таблицы динамического списка с пересечением колонок на одной форме» редкий; при необходимости передавайте имена колонок явно через `fields`.

## Права ролей: setRoleRight теперь реально пишет в Rights.rights

### Тихий сбой записи прав роли на диск

**Как было.** Агент вызывает `setRoleRight` на роли — например, «дай Read на `Catalog.Номенклатура`». Плагин отвечает `success: true`, `applied: "1 right(s) on Catalog.Номенклатура: Read=SET"`. Никаких ошибок. Агент считает, что всё хорошо, и переходит к следующему шагу. А в пользовательском режиме право не работает — потому что файл `<Роль>/Rights.rights` на диске не обновился. Обнаруживается это только после публикации на базу или когда разработчик открывает роль в EDT и не видит там проставленного права. Единственный workaround — ручная правка `Rights.rights` и обновление проекта.

**Причина.** У роли в 1С:EDT описание прав (`RoleDescription`) — это **отдельный объект**, который хранится в собственном файле `Rights.rights` рядом с `<Роль>.mdo`. Плагин корректно правил этот объект внутри базы модели (BM) и даже регистрировал его там с правильным идентификатором. Но на этапе записи изменений на диск (batch-forceExport после операции) в список файлов для сохранения попадало только само имя роли — `<Роль>.mdo` — и не попадал объект-описание прав `<Роль>.Rights`. Поэтому EDT сохранял в памяти всё корректно, а на диск — нет. Срабатывало молча, без ошибки.

**Что стало.** В список файлов для сохранения после `setRoleRight` добавлен описательный объект роли. Теперь при любом вызове — установить право, снять право, пакет `rights: [...]` с несколькими правами сразу — файл `Rights.rights` переписывается на диске в том же вызове, что и ответ агенту. Никакой ручной правки не требуется, EDT-редактор роли сразу показывает актуальные права.

В ответе плагина теперь видно оба сохранённых файла: `exportSync.fqns: ["Role.<Имя>", "Role.<Имя>.Rights"]`. Если `.Rights` нет в этом списке — скорее всего запущена старая версия плагина, нужно обновиться.

Поведение одинаковое для всех трёх значений: `SET` (право разрешено), `UNSET` (явно запрещено), `PROVIDED` (унаследовано). В файле `Rights.rights` при `UNSET` появляется `<value>false</value>`, при `SET` — `<value>true</value>`.

Заодно параметр `targetObject` теперь принимает **путь к вложенному элементу**, а не только объект целиком. Можно ставить право на отдельный реквизит (`Catalog.Клиенты.Attribute.Паспорт`), табличную часть (`Document.Заказ.TabularSection.Товары`), колонку ТЧ (`Document.Заказ.TabularSection.Товары.Attribute.Цена`), команду (`Catalog.Товары.Command.ОткрытьОтчёт`), измерение/ресурс регистра (`InformationRegister.Курсы.Dimension.Валюта`). Если для указанной цели платформа не предусматривает прав (например, значение перечисления) — плагин вернёт ошибку `"No available rights for target object"`, а не молчаливый успех.
