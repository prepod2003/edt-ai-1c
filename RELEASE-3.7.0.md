## Запись DCS-схем в расширениях — главный фикс

**Как было.** Агент создаёт отчёт в расширении конфигурации через `createReportSchema`, дальше `addDataSet`, `addCalculatedField`, `addTotalField` и так далее — каждый вызов отвечает `success:true`, в редакторе СКД EDT превью схемы выглядит правильно, всё на месте. Но при перезапуске EDT превью становится пустым, а попытка обновить конфигурацию базы данных падает с ошибкой 1С:Предприятия «Неизвестный объект метаданных - Report.X.Template.ОсновнаяСхемаКомпоновкиДанных». Все правки агента теряются. То же самое для DataProcessor с DCS-макетом и для внешних обработок-отчётов.

**Причина.** В EDT 2025.2 есть баг внутреннего сериализатора DCS-схемы: при работе в проекте расширения первая запись `.dcs` на диск проходит, дальше — молча перестаёт работать. Файл на диске не обновляется (или вовсе не создаётся), но в оперативной памяти EDT схема остаётся правильной — поэтому редактор СКД показывает её корректно, и агент тоже не видит проблемы. Боль вылезает только при следующем запуске EDT (память перечитывается с диска — там пусто) или при deploy в инфобазу (1С смотрит на диск, а не на EDT — там тоже пусто).

**Что стало.** Плагин больше не полагается на штатный путь записи `.dcs` в EDT — он сам сериализует DCS-схему в свой файл на диске сразу после изменения в памяти. Это происходит прозрачно: никаких новых параметров не нужно, любой вызов `edit_metadata` для СКД в расширении (`createReportSchema`, `addDataSet`, `addSchemaParameter`, `addCalculatedField`, `addTotalField`, `setSettingsParameter`, `addConditionalAppearance` и любая другая DCS-операция) теперь физически пишет файл `.dcs` на диск.

В ответе появилось новое поле `exportSync.directDcs` с диагностикой по каждой записанной схеме:

```json
"exportSync": {
  "fqns": ["Report.МойОтчёт"],
  "directDcs": [{
    "fqn": "Report.МойОтчёт.Template.ОсновнаяСхемаКомпоновкиДанных.Template",
    "ok": true,
    "filePath": "\\МоёРасширение\\src\\Reports\\МойОтчёт\\Templates\\ОсновнаяСхемаКомпоновкиДанных\\Template.dcs",
    "bytes": 2315,
    "ms": 40
  }]
}
```

Покрытие универсальное — работает для отчётов (`Report`), обработок (`DataProcessor`), внешних отчётов и обработок (`ExternalReport`, `ExternalDataProcessor`) и других объектов с СКД-макетом. Для основной конфигурации фикс срабатывает тоже — там этот же баг проявлялся реже, но мог давать тихие потери записей в свежих workspace.

## Лечение унаследованных битых отчётов — `repairReportSchema`

**Что это и зачем.** Если у вас в расширении остались отчёты или обработки с СКД, которые делались через предыдущие версии плагина и из-за описанного выше бага записи остались «фантомами» (`.mdo`-файл есть, ссылается на DCS-макет, а самого `.dcs` файла на диске нет или он пустой), — теперь их можно вылечить одним вызовом без ручного пересоздания. Операция читает существующий `.dcs` с диска и подставляет схему на место в EDT. Идемпотентна: если схема в памяти EDT уже корректная — ничего не меняет, возвращает `alreadyValid: true`.

**Как пользоваться.**

```json
{"operation":"repairReportSchema","projectName":"МоёРасширение",
 "objectName":"Report.БитыйОтчёт"}
```

Параметр `templateName` опциональный (по умолчанию `ОсновнаяСхемаКомпоновкиДанных`) — указывайте, если макет не основной. В ответе вернутся `success`, `schemaFqn`, `readFromFile` (путь прочитанного файла), `readBytes` и `readMs`. Если `.dcs` на диске тоже отсутствует или повреждён настолько, что прочитать нечего — операция вернёт явную ошибку «.dcs файл не найден на диске», и тогда единственный путь — `removeObject` + создать схему заново.

После апдейта плагина новые битые фантомы создаваться больше не будут (см. главный фикс выше) — операция `repairReportSchema` нужна только для лечения уже накопленных в проекте.

## Условное оформление: понятная ошибка вместо порчи XML при Font-объекте

**Как было.** В операциях `addConditionalAppearance` и `setDataSetFieldAppearance` параметр `appearance` принимает массив `[{name, value}]`. Если агент передавал значение шрифта или цвета как JSON-объект (например `{"name":"Шрифт","value":{"bold":true,"size":12}}`) — плагин не возвращал ошибку, а тихо записывал такое значение в `.dcs` в виде сломанного XML-атрибута. Дальше редактор СКД в EDT не мог открыть схему, превью становилось пустым.

**Что стало.** Если значение параметра оформления передано как JSON-объект или массив — плагин сразу возвращает понятную ошибку с инструкцией правильного формата:

```text
Параметр оформления 'Шрифт' получен как JSON-объект/массив. Передавай
значение строкой: для шрифта — "Arial,12,bold" (имя,размер,флаги:
bold/italic/underline); для цвета — "#RRGGBB" или имя стиля. JSON-объекты
типа {"bold":true} в текущей версии плагина не поддержаны.
```

Никакой записи на диск при этом не происходит — `.dcs` остаётся валидным. Правильный формат значений уже описан в `edit_metadata operation=help topic=dcsAppearance` (шрифт — `'Arial,12,bold'`, цвет — `'#RRGGBB'`).

## Новые операции для СКД-схемы

### Удаление набора данных — `removeDataSet`

Раньше через плагин нельзя было удалить лишний набор данных из СКД-схемы — приходилось править руками. Теперь:

```json
{"operation":"removeDataSet","projectName":"МойПроект",
 "objectName":"Report.X","name":"ЛишнийНабор"}
```

Идемпотентна — повторный вызов на уже удалённый набор вернёт `notFound:true` без ошибки.

### Изменение параметра СКД — `setSchemaParameter`

Если параметр уже создан, но нужно поменять тип, длину, выражение по умолчанию или сделать его множественным (списком значений) — раньше для этого не было операции, агент мог только удалить старый и создать новый. Теперь:

```json
{"operation":"setSchemaParameter","projectName":"МойПроект",
 "objectName":"Report.X","name":"ДатаНачала","type":"Date"}
```

Поддерживаемые поля для изменения: `type`, `length`, `precision`, `valueListAllowed`, `denyIncompleteValues`, `expression`, `title`, `use`. Меняются только переданные параметры — остальные сохраняются.

### Удаление и перестановка параметра — `removeSchemaParameter`, `moveSchemaParameter`

```json
{"operation":"removeSchemaParameter","projectName":"МойПроект",
 "objectName":"Report.X","name":"ЛишнийПараметр"}

{"operation":"moveSchemaParameter","projectName":"МойПроект",
 "objectName":"Report.X","name":"ДатаНачала","direction":"Up"}
```

`moveSchemaParameter` принимает либо `direction` (`Up`/`Down`), либо `newIndex` (0-based позиция в списке).

### Параметр в шапку отчёта — `viewMode=QuickAccess`

Чтобы пользователь видел параметр прямо в шапке отчёта (быстрые настройки на форме отчёта), а не лез в полный редактор настроек СКД:

```json
{"operation":"setSettingsParameter","projectName":"МойПроект",
 "objectName":"Report.X","name":"Партнёр","viewMode":"QuickAccess"}
```

Допустимые значения `viewMode`: `Auto`, `Normal`, `QuickAccess`, `Inaccessible` (русские алиасы — `Авто`, `Обычный`, `БыстрыйДоступ`, `Недоступен`). По умолчанию `Auto`.

## Авто-валидация СКД до записи

### Текст запроса проверяется перед записью

В операции `addDataSet` с `dataSetType=Query` плагин теперь сам прогоняет переданный `queryText` через парсер запросов перед записью в схему. Самые частые грабли, которые раньше попадали в `.dcs` и вылезали уже в редакторе СКД EDT — `{ГДЕ ...}` в неверном месте, `ЗначениеЗаполнено(&Параметр)` в WHERE, незакрытые скобки — теперь блокируют операцию сразу:

```json
"queryValidation": {
  "issues": [
    {"line": 9, "column": 5, "severity": "ERROR",
     "message": "Синтаксическая ошибка. Недопустимая лексема `{`"}
  ],
  "statistics": {"errors": 1, "warnings": 0}
}
```

Если запрос валидный — операция проходит как раньше. Поведение можно отключить параметром `validateQueryBeforeWrite=false` для редких конструкций, не известных парсеру (например, `Регистр.X.Обороты(...)` со специфичными виртуальными таблицами).

### Выражения вычисляемых полей и итогов проверяются перед записью

То же самое для выражений СКД — `addCalculatedField` (`expression`), `addTotalField` (`expression`) и `addSchemaParameter`/`setSchemaParameter` (`expression`). Опечатки в имени функции (`Сумм` вместо `Сумма`), несбалансированные скобки и другие синтаксические ошибки выражений ловятся до записи в схему. Параметр `validateExpressionBeforeWrite=false` отключает проверку для крайних случаев.
