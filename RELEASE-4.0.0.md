# MCP:RSV Server — Релиз 4.0.0

С этого релиза AI-агент в 1С:EDT умеет не только писать код и менять метаданные, но и **сам прогонять юнит-тесты, и сам разбираться с упавшими через отладчик**. Цикл «правка кода → прогон тестов → пошаговая отладка → результат» теперь замыкается без человека: ни одного клика мышью между правкой и отчётом.

Это новая компетенция плагина, поэтому версия мажорная. Старые инструменты (`edit_metadata`, `write_module_source`, `launch_debugger`, `export_object` и все остальные) продолжают работать как раньше — ничего не сломано, ничего не переименовано. Просто добавилась новая большая возможность.

## Юнит-тесты YAxUnit — закрытый цикл AI-разработки

YAxUnit — это бесплатный фреймворк юнит-тестов для 1С (Apache 2.0, [github.com/bia-technologies/yaxunit](https://github.com/bia-technologies/yaxunit)), де-факто стандарт в сообществе. Подключается к любой информбазе как расширение конфигурации (`.cfe`).

В этом релизе появилась полная интеграция YAxUnit в плагин. Раньше, чтобы прогнать тесты, программист должен был открыть в 1С:Предприятии специальную обработку, нажать «Запустить», дождаться результата, посмотреть отчёт глазами, перейти к коду, поправить баг, повторить. Теперь это делает AI-агент в один вызов, а если тест упал — сам же открывает отладчик и доходит до места, где сломалось.

### Запуск тестов одним вызовом — `yaxunit_tests`

**Что это и зачем.** Новый инструмент `yaxunit_tests` запускает юнит-тесты YAxUnit в любой настроенной информбазе проекта и возвращает агенту готовый Markdown-отчёт: сколько тестов прошло, сколько упало, сколько времени заняло, для упавших — сообщение об ошибке и стек вызовов. Никаких ручных операций в 1С:Предприятии — агент сам стартует приложение со специальным режимом «прогон тестов», ждёт окончания, читает результат и форматирует его.

Поддерживаются фильтры — можно прогнать только конкретное расширение, конкретные общие модули с тестами, конкретные тесты по имени, по тестовым наборам, по тегам или контекстам исполнения (Server / Client / ExternalConnection).

**Как пользоваться.**

```json
{"projectName":"МойПроект","applicationId":"<id>","modules":"ПростыеТесты"}
```

В ответе — Markdown-отчёт + структурированная сводка (`totalTests`, `passed`, `failed`, `errors`, `skipped`, `durationSeconds`). Если прогон длинный и не успел в окно ожидания — `yaxunit_tests` **не убивает** 1С, а возвращает `status=Pending`. Повторный вызов с теми же параметрами заберёт результат когда тот появится. Никаких ручных таймаутов и кнопок «остановить».

Требует, чтобы у проекта была настроена Run Configuration в EDT (Run → Run Configurations → 1C:Enterprise) и расширение YAxUnit было подключено к информбазе (см. ниже встроенный топик `help=setup`).

### Отладка упавшего теста — `mode=debug` и точки останова

**Что это и зачем.** Когда тест упал, понять причину «по сообщению об ошибке» получается не всегда. Раньше это означало переключиться в 1С:Предприятие, поставить точку останова, перезапустить тест в режиме отладки, дождаться срабатывания, посмотреть переменные. Теперь то же самое делается полностью внутри AI-агента.

`yaxunit_tests` с параметром `mode=debug` запускает 1С в режиме отладки. Точки останова, поставленные заранее через `launch_debugger action=addBreakpoint`, срабатывают как обычно. Когда тест дошёл до точки останова, `yaxunit_tests` возвращает `status=Pending` (ничего не убивает), а через `launch_debugger` агент изучает состояние: смотрит стек, текущие значения переменных, вычисляет произвольные BSL-выражения, делает шаг или продолжает выполнение.

**Как пользоваться.**

```json
{"action":"addBreakpoint","modulePath":"CommonModules/ПростыеТесты/Module.bsl","lineNumber":58}
```

```json
{"projectName":"МойПроект","applicationId":"<id>","tests":"ПростыеТесты.Тест_РазмерСкидки","mode":"debug"}
```

Дальше — `launch_debugger getState` (увидеть на какой строке остановились), `getVariables` (текущие значения), `evaluate` (проверить «а что если посчитать по-другому?»), `resume` (продолжить). После завершения прогона повторный вызов `yaxunit_tests` с теми же параметрами заберёт финальный отчёт.

### Автообновление информбазы перед запуском

**Что это и зачем.** Когда AI-агент только что поправил код через `write_module_source`, информбаза остаётся со старой версией кода — ровно как если бы вы сами поправили модуль в EDT и запускали 1С:Предприятие. В таком случае платформа показывает модальное окно *«Обновить конфигурацию информационной базы?»* с кнопками «Обновить и запустить» / «Запустить без обновления» / «Отменить».

AI-агент мышью кликнуть не может — окно блокирует автономный цикл. Поэтому `yaxunit_tests` по умолчанию **сам** синхронизирует информбазу с проектом перед запуском, ровно как пункт меню Run → Update → Update Database. Никакого диалога, тесты сразу прогоняются на свежей версии.

**Как пользоваться.** Параметр `updateBeforeLaunch` включён (`true`) по умолчанию — ничего делать не надо. Если в редком случае хочется пропустить обновление (например, вы вручную сделали `update_database` минуту назад) — передайте `updateBeforeLaunch=false`. Если синхронизация не удалась — например, потому что на этой же информбазе ещё работает запущенный 1С:Предприятие или другой EDT держит файл блокировки — `yaxunit_tests` вернёт внятную ошибку с описанием причины и подсказками что проверить, а не молча зависнет.

### Встроенная справка для AI — `help=topics` и пять тем

**Что это и зачем.** В голове AI-агента, особенно если это слабая или младшая модель, может вообще не быть знаний про синтаксис YAxUnit — что такое `ИсполняемыеСценарии`, как зарегистрировать тест через `ЮТТесты.ДобавитьСерверныйТест`, какие бывают проверки в `ЮТест.ОжидаетЧто(...)`. Без справки агент напишет «обычный» BSL и удивится «прогон отчитал — 0 тестов». Поэтому `yaxunit_tests` несёт встроенную справку: агент по запросу подгружает короткий тематический туториал, не перегружая контекст.

Доступные темы:

- `topics` — список всех тем с одной строкой описания каждой;
- `writing` — как написать первый тест: общий модуль + `ИсполняемыеСценарии` + AAA-схема (Arrange/Act/Assert) + минимальный шаблон;
- `assertions` — каталог проверок: `Равно`, `НеРавно`, `Содержит`, `Заполнено`, `Имеет`, `ВыбрасываетИсключение` и др., с примерами;
- `setup` — как подключить расширение YAxUnit (`.cfe`) к информбазе;
- `events` — `ПередКаждымТестом`, `ПослеТестовогоНабора`, контексты теста/набора/модуля;
- `advanced` — указатели на публичную документацию YAxUnit для тех тем, что не покрыты встроенной справкой (моки, стабы, RPC-режим, проверки против информбазы).

**Как пользоваться.**

```json
{"help":"writing"}
```

При `help=<тема>` остальные параметры игнорируются — это чисто справочный вызов, тесты не запускаются. Возвращается Markdown с шаблоном или каталогом. Полезно даже сильным моделям — это всё равно быстрее, чем гуглить.

### Подсказка, когда не нашли ни одного теста

**Что это и зачем.** Самая частая ошибка при первой работе с YAxUnit — написать процедуры с правильными именами (`Тест_СложениеДваПлюсДва`), но забыть зарегистрировать их в экспортной процедуре `ИсполняемыеСценарии`. YAxUnit запускает только то, что явно зарегистрировано — без регистрации тест в отчёте не появляется. Агент видит «прогон отчитал, 0 тестов» и не понимает, что не так.

В этом релизе Markdown-отчёт `yaxunit_tests` при нулевом результате теста выдаёт **развёрнутую подсказку**: три самые частые причины (тесты не зарегистрированы в `ИсполняемыеСценарии`; фильтры не совпадают с реальными именами; YAxUnit не подключён к информбазе) и ссылку на встроенную справку — `yaxunit_tests help=writing` для шаблона минимального теста или `yaxunit_tests help=setup` для инструкции по подключению расширения. Дальше — агент сам исправляет проблему и повторяет прогон.
