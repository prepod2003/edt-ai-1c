## Совместимость с AI-инструментами

### Codex (расширение OpenAI для VS Code) — теперь подключается

**Как было.** При попытке подключить плагин к Codex (это AI-агент от OpenAI — отдельное расширение для VS Code, входит в подписку ChatGPT Plus / Pro / Go) подключение падало с ошибкой, и агент Codex плагин не видел вообще. Ни один инструмент из плагина — ни чтение кода, ни редактирование метаданных — Codex использовать не мог. С другими AI-инструментами (Claude Code, Cursor, Roo Code, Cline, Windsurf и т. д.) плагин работал нормально, поэтому проблема всплыла не сразу.

**Причина.** Способ, которым плагин «представлялся» AI-инструменту при первом подключении, подходил для одних агентов и не подходил для других — более строгих. Claude Code и Cursor пропускали его без вопросов, а Codex по строгой проверке отказывался работать и закрывал соединение. Кроме того, в части ответов плагина не хватало служебной пометки, которую современные AI-агенты считают обязательной — без неё они просто не понимают, что им вернули.

**Что стало.** Плагин теперь отвечает агенту в формате, который понимают все распространённые AI-инструменты — и старые, и новые. Codex (VS Code Extension) подключается обычным способом — через тот же файл настроек `~/.codex/config.toml`, что и любой другой MCP-сервер. На стороне пользователя ничего делать не нужно — кроме обновления плагина до этой версии. Поведение для уже работавших инструментов (Claude Code, Cursor, Roo Code, Cline, Windsurf, Gemini, Kimi CLI, Qwen Code) не изменилось — они продолжают работать как раньше.

Если ты пользуешься Codex и заходишь в него через подписку ChatGPT (а не через ключ API), важно: модели с пометкой `*-codex` (например `gpt-5-codex`) у такого входа недоступны — это ограничение OpenAI, не плагина. В конфиге Codex `~/.codex/config.toml` строку `model = "gpt-5-codex"` лучше убрать вовсе — тогда Codex сам подберёт совместимую модель.

## Отчёты СКД — что нового

Главное изменение релиза. Раньше плагин умел собрать только простейшую схему компоновки данных — фактически «болванку» отчёта, которую дальше приходилось дорабатывать руками в редакторе EDT. Теперь редактировать схему отчёта через MCP можно так же полно, как в самом редакторе СКД: всё, что отчёт умеет в платформе — наборы данных, связи, параметры, вычисляемые поля, итоги, группировки, кросс-таблицы, диаграммы, отборы, сортировки, условное оформление, варианты — собирается по одной просьбе агента, без открытия редактора вручную.

Всего по отчётам СКД доступно **48 операций**. Полный перечень с кратким описанием каждой — в разделе [«Полный список операций по отчётам СКД»](#полный-список-операций-по-отчётам-скд). Ниже — что именно стало возможно в этом релизе.

### Полное редактирование схемы, а не только сборка с нуля

В прошлом релизе почти всё в схеме можно было только **добавить** — изменить или удалить уже добавленное было нельзя. Если агент ошибся или пользователь передумал, приходилось открывать редактор СКД руками.

Теперь у каждого элемента схемы есть полный набор действий — создать, изменить, удалить: для наборов данных и их полей, параметров, вычисляемых полей, итогов, отборов, сортировок, условного оформления и вариантов отчёта. Это значит, что агент может править готовый отчёт точечно — переименовать набор, поменять формулу итога, убрать лишний отбор — а не пересобирать схему целиком.

### Связи наборов данных — теперь можно менять и удалять, а не только создавать

В отчёте часто собирают данные из нескольких источников и связывают их между собой — типичный пример: продажи в одном наборе, реквизиты организаций (ИНН, КПП, адрес) в другом, связаны по полю «Организация». Раньше связь можно было только создать; прочитать, поправить или удалить — нет.

Теперь связь можно создать, изменить и удалить. Плюс плагин сам поддерживает порядок: при удалении набора висящие на нём связи убираются автоматически, а при переименовании набора связи переключаются на новое имя — без ручной чистки.

### Варианты отчёта — полноценное управление

Отчёт может иметь несколько вариантов настроек — например, «Продажи по организациям» и «Продажи по номенклатуре». Раньше вариант можно было только добавить — переименовать, удалить или скопировать существующий было нельзя. А все настройки (группировки, отборы, выбранные поля) всегда попадали в основной вариант, даже если нужен был другой.

Теперь вариант можно добавить, скопировать целиком (со всеми группировками, отборами и оформлением), переименовать и удалить. А любую настройку можно адресовать конкретному варианту — указав, в какой из них её добавить. Многовариантные отчёты собираются полностью через MCP, без открытия редактора.

### Правка запроса набора без переписывания всего текста

Раньше, чтобы добавить в запрос набора одно поле или одно условие, приходилось передавать весь текст запроса заново — иначе терялись остальные части (отборы, упорядочивание, группировки). Агент был обязан держать в памяти весь запрос целиком.

Теперь можно добавить одно поле в выборку, убрать одно поле или добавить одно условие отбора — остальной запрос остаётся нетронутым.

### Быстрые настройки в шапке отчёта

В отчёте часть настроек обычно показывают пользователю прямо в шапке («быстрые настройки» — период, переключатель оформления, фильтр организации), а часть прячут в форму «Изменить вариант...». Раньше управлять этим через MCP было нельзя — все настройки выглядели одинаково.

Теперь для любой настройки (отбор, сортировка, выбранное поле, оформление, параметр) можно указать, показывать ли её в шапке отчёта или спрятать. То есть агент сразу собирает отчёт с удобной шапкой — пользователю не нужно лезть в настройки варианта.

### Внешние наборы данных — рабочая заготовка кода вместо пустого скелета

Бывает, что данные для отчёта берутся не из запроса, а готовятся в коде (внешний набор данных — таблица значений, которую заполняет процедура отчёта). Раньше плагин вписывал в модуль только пустую заготовку процедуры — заполнять таблицу, описывать колонки приходилось вручную.

Теперь заготовка рабочая: в ней уже создаётся таблица значений, прописаны её колонки под добавленные поля и выполнено подключение таблицы к отчёту — остаётся только заполнить её данными. Более того, когда агент добавляет в такой набор новое поле, плагин **сам** дописывает соответствующую колонку в код — раньше это поле молча «терялось», и отчёт падал в Предприятии с ошибкой о ненайденном поле.

### Предпросмотр в редакторе СКД обновляется сразу

Раньше после каждого изменения схемы через MCP открытый в EDT редактор СКД продолжал показывать старое состояние — пока пользователь не перезапускал EDT или не переоткрывал макет. На отчётах из десятков правок это было главным неудобством.

Теперь предпросмотр в открытом редакторе СКД обновляется автоматически после каждой операции — ни перезапускать EDT, ни переоткрывать макет не нужно, изменения видны сразу. (Есть одно исключение для запросных наборов во внешних отчётах — см. [«Известные ограничения»](#известные-ограничения).)

### Внешние отчёты — наравне с обычными

Внешние отчёты (`.erf`) теперь поддержаны в полном объёме — так же, как обычные отчёты в конфигурации. Раньше у них работало не всё: например, форма не могла корректно обратиться к самому внешнему отчёту, и часть платформенных конструкций в рантайме возвращала пусто. Сейчас внешний отчёт собирается целиком — со схемой компоновки данных, формой, реквизитами и рабочим обращением к настройкам отчёта.

Кроме того, при проверке запросов внешнего отчёта плагин теперь видит таблицы той конфигурации, к которой отчёт подключён. Раньше запрос к регистру или справочнику основной конфигурации помечался как ошибочный («таблица не найдена»), хотя на самом деле был правильным и в Предприятии работал. Теперь ложных ошибок нет.

### Сборка большого отчёта стала надёжной

Сложный отчёт — это десятки операций подряд. На таком объёме раньше возникали неприятные эффекты, которые этот релиз закрывает:

- **Повторный вызов больше не плодит дубли.** Если связь с агентом оборвалась и он повторил шаг, раньше в схеме мог появиться второй параметр (вариант, оформление) с тем же именем — и отчёт потом не открывался в Предприятии с ошибкой о повторяющемся имени. Теперь повторный вызов с тем же именем ничего не дублирует и честно сообщает агенту, что элемент уже есть.
- **Агент видит схему чужого отчёта целиком до правки.** При сборе сведений об отчёте плагин теперь отдаёт полный обзор схемы — наборы данных и их поля, связи, параметры, вычисляемые поля, итоги, варианты, оформление. Раньше агент работал почти вслепую и узнавал состояние схемы только методом проб.
- **Отчёт можно собрать одним пакетным вызовом.** Вместо десятков отдельных обращений к плагину агент отправляет список операций разом — плагин выполняет их по очереди и возвращает общий итог. Это быстрее и заметно легче для AI-моделей с небольшим контекстом.

### Что ещё добавилось

Мелкие, но полезные операции, дополняющие набор: перестановка параметров схемы в нужном порядке, очистка всего списка выводимых полей одной командой, настройка параметров вывода (заголовок отчёта, тип диаграммы, макет оформления), пользовательские поля-выражения и восстановление схемы (перечитать с диска, если она оказалась в рассинхроне).

## Полный список операций по отчётам СКД

Полный набор — **48 операций**, сгруппированных по смыслу. Этот раздел можно использовать как справочник: что вообще умеет плагин по отчётам СКД.

### Схема отчёта

| Операция | Что делает |
|---|---|
| `createReportSchema` | Создать пустую схему компоновки данных в макете объекта |
| `repairReportSchema` | Перечитать схему с диска и обновить её в проекте (восстановление при рассинхроне) |

### Наборы данных

| Операция | Что делает |
|---|---|
| `addDataSet` | Добавить набор данных: запрос, объект (данные из кода) или объединение |
| `setDataSetProperty` | Изменить набор: имя, источник, текст запроса |
| `removeDataSet` | Удалить набор данных по имени |

### Поля набора данных

| Операция | Что делает |
|---|---|
| `addDataSetField` | Добавить поле в набор данных |
| `removeDataSetField` | Удалить поле из набора данных |

### Правка запроса набора

| Операция | Что делает |
|---|---|
| `addQueryField` | Добавить одно поле в выборку запроса, не переписывая запрос целиком |
| `removeQueryField` | Убрать поле из выборки запроса |
| `addQueryCondition` | Добавить условие в отбор запроса |

### Связи наборов данных

| Операция | Что делает |
|---|---|
| `addDataSetLink` | Связать два набора данных между собой |
| `setDataSetLinkProperty` | Изменить существующую связь |
| `removeDataSetLink` | Удалить связь |

### Параметры схемы

| Операция | Что делает |
|---|---|
| `addSchemaParameter` | Добавить параметр схемы |
| `setSchemaParameter` | Изменить параметр схемы |
| `removeSchemaParameter` | Удалить параметр схемы |
| `moveSchemaParameter` | Переставить параметр на другую позицию в списке |

### Вычисляемые поля

| Операция | Что делает |
|---|---|
| `addCalculatedField` | Добавить вычисляемое поле (поле-формулу) |
| `setCalculatedField` | Изменить вычисляемое поле |
| `removeCalculatedField` | Удалить вычисляемое поле |

### Итоги (ресурсы)

| Операция | Что делает |
|---|---|
| `addTotalField` | Добавить итог — агрегированное поле (сумма, количество и т. п.) |
| `setTotalField` | Изменить выражение или группировки итога |
| `removeTotalField` | Удалить итог |

### Структура вывода — группировки, таблицы, диаграммы

| Операция | Что делает |
|---|---|
| `addSettingsGroup` | Добавить группировку (в том числе вложенную или внутри таблицы/диаграммы) |
| `addSettingsTable` | Добавить кросс-таблицу (строки × колонки) |
| `addSettingsChart` | Добавить диаграмму (точки × серии) |
| `removeSettingsItem` | Удалить группировку, таблицу или диаграмму |

### Выводимые поля

| Операция | Что делает |
|---|---|
| `addSettingsSelectedField` | Добавить поле в список выводимых полей отчёта |
| `removeSettingsSelectedField` | Убрать поле из списка выводимых |
| `clearSettingsSelectedFields` | Очистить весь список выводимых полей |

### Отборы

| Операция | Что делает |
|---|---|
| `addSettingsFilter` | Добавить отбор |
| `addSettingsFilterGroup` | Добавить группу отборов (И / ИЛИ / НЕ) |
| `removeSettingsFilter` | Удалить отбор |

### Сортировки

| Операция | Что делает |
|---|---|
| `addSettingsOrder` | Добавить сортировку |
| `removeSettingsOrder` | Удалить сортировку |

### Условное оформление

| Операция | Что делает |
|---|---|
| `addConditionalAppearance` | Добавить условное оформление (цвет, шрифт, формат, границы, выравнивание) |
| `setConditionalAppearance` | Изменить условное оформление |
| `removeConditionalAppearance` | Удалить условное оформление |
| `setDataSetFieldAppearance` | Задать постоянное оформление поля набора (например, формат числа) |

### Варианты отчёта

| Операция | Что делает |
|---|---|
| `addSettingsVariant` | Добавить вариант отчёта |
| `cloneSettingsVariant` | Скопировать вариант целиком под новым именем |
| `setSettingsVariantProperty` | Переименовать вариант или сменить его заголовок |
| `removeSettingsVariant` | Удалить вариант (последний удалить нельзя) |

### Значения параметров и параметры вывода

| Операция | Что делает |
|---|---|
| `setSettingsParameter` | Задать значение параметра в настройках отчёта |
| `removeSettingsParameter` | Снять значение параметра |
| `setOutputParameter` | Задать параметр вывода: заголовок отчёта, тип диаграммы, макет оформления |

### Быстрые настройки и пользовательские поля

| Операция | Что делает |
|---|---|
| `setSettingsItemUserMode` | Вынести настройку (отбор, сортировку, оформление, группировку, поле) в шапку отчёта |
| `addUserField` | Добавить пользовательское поле-выражение |

> Итого: **16 групп, 48 операций.** Кроме того, при сборе сведений об объекте через `ai_context` плагин отдаёт полный обзор схемы отчёта (наборы, поля, связи, параметры, итоги, варианты, оформление) — агент видит состояние схемы до того, как начать правку.

## Известные ограничения

### Предпросмотр запросных наборов в проектах внешних отчётов

Когда во внешнем отчёте (отдельный проект `.erf` в EDT) добавляется набор данных на основе **запроса**, предпросмотр схемы в самом редакторе СКД остаётся **пустым** до тех пор, пока пользователь не откроет макет схемы руками первый раз. Это особенность самой EDT, а не плагина: для внешних отчётов поля запросного набора вычисляются редактором только при первом открытии макета, и подменить этот шаг извне нельзя.

На практике это значит:

- В обычной конфигурации (отчёт в составе конфигурации) ограничения нет — предпросмотр работает сразу.
- Для наборов «из кода» (объектных) в проекте внешнего отчёта ограничения нет — предпросмотр тоже работает сразу.
- Для запросных наборов во внешнем отчёте: собранный `.erf` запускается в 1С:Предприятии нормально, поля подтянутся автоматически при первом открытии в платформе. Увидеть предпросмотр **именно в редакторе СКД EDT** можно, один раз открыв макет вручную.

Это ограничение отражено в справке плагина — чтобы AI-агент не пугался пустого предпросмотра и доводил сборку отчёта до конца.
