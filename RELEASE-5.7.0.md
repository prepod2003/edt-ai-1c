## Запись кода

### Новый модуль больше не создаётся в чужом проекте

Если модуль объекта ещё не существовал на диске, `write_module_source` мог создать
его файл не в том проекте: без `projectName` инструмент брал первый попавшийся
1С-проект workspace, и, например, менеджерский модуль документа основной
конфигурации мог оказаться в DT-проекте внешнего отчёта, где такого объекта нет
вовсе. Файл-сирота оставался вне модели EDT — невидимый и для платформы, и для
последующих операций.

Теперь при создании нового файла проект определяется по объекту-владельцу модуля,
а не «первый попавшийся». Если объект не найден ни в одном проекте workspace —
запись отклоняется с понятной ошибкой (проверьте имя объекта или сначала создайте
его через `edit_metadata createObject`) вместо молчаливого создания сироты. Если
`projectName` указан явно, но объекта в этом проекте нет — запись тоже отклоняется.
В ответе (и в `dryRun`) теперь есть поле `projectName` — агент видит, в какой
проект реально пойдёт файл.

## Обновление информационной базы

### Незавершённая сессия объединения больше не выдаётся за успех

Если в EDT осталась незавершённая сессия объединения с изменениями информационной
базы (после ручного обновления базы конфигуратором или прерванной полной загрузки),
`sync_database` мог вернуть `success` за одну-две секунды, ничего не обновив;
дальше по цепочке headless-запуск тестов падал с ошибкой «Проект сейчас находится
в процессе объединения…».

Теперь обновление ИБ (общий путь для `sync_database`, `yaxunit_tests` и
`launch_debugger`) заранее распознаёт активную сессию объединения и возвращает
явную ошибку с подсказкой завершить или отменить её в EDT. Дополнительно, если
обновление всё же отчиталось успехом, но состояние базы осталось «требуется
обновление», в ответ добавляется предупреждение с тем же указанием — молчаливого
«успеха без результата» больше нет.
