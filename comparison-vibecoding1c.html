1C:RSV MCP Server и vibecoding1c.ru
— два подхода к AI-разработке на 1С

Технический разбор двух MCP-инструментов для AI-разработки на 1С: архитектура, возможности, сценарии применения. Чтобы помочь выбрать инструмент под конкретные задачи.

Несколько слов о коллегах

Команда vibecoding1c.ru проделала большую и важную работу. Они одними из первых на российском рынке построили целую экосистему для интеграции AI с 1С: семь MCP-серверов, RAG-индексы, граф связей в Neo4j, база из 2500+ шаблонов, обширные cursor-rules для агентов, видеокурсы, статьи, активное сообщество в Telegram.

Это серьёзный труд, и их вклад в популяризацию темы AI-разработки на 1С трудно переоценить. Поэтому это сравнение — не «кто лучше», а разговор о двух разных архитектурных подходах к одной и той же задаче.

Главное в одном абзаце

У обоих подходов есть своя логика, свои сильные стороны и свои ограничения. Ниже — суть каждого:

vibecoding1c

7 RAG-серверов вокруг Cursor

Семь Docker-контейнеров: RAG по метаданным, по коду, по справке платформы и БСП, граф Neo4j, обёртка над BSL Language Server, интеграция с 1С:Напарник. Это поисковики контекста — они помогают AI найти и понять, как сделать. Дальше Cursor редактирует .bsl-файлы и XML как обычный текст.

1C:RSV

Один плагин внутри 1С:EDT

Сам плагин становится MCP-сервером для любого AI-клиента. AI не правит файлы — он вызывает операции, а EDT выполняет их через свои штатные сервисы (EMF-модель, конструкторы метаданных, парсер Xtext, валидатор, Eclipse Debug API). Любое действие AI идентично ручным действиям разработчика в EDT. Три профиля — Аналитик, Разработчик, Архитектор — закрывают весь спектр задач: от изучения любой конфигурации до её модификации.

Это ключевое архитектурное различие порождает все остальные. Для одних задач удобнее первый подход, для других — второй.

Принципиальная разница: XML-файлы или нативный API EDT

Это центральный технический вопрос, и от него зависит почти всё остальное.

Файлы конфигурации 1С:EDT (.mdo, .form, .bsl, XML дочерних элементов) — это сериализация EMF-модели Eclipse. У каждого объекта свой UUID, перекрёстные ссылки между объектами, упорядоченные коллекции, метаинформация платформы. Когда AI редактирует такие файлы как текст, он берёт на себя ответственность за корректность всей этой структуры.

Для простых правок BSL-кода это нормально: модуль — это плоский текст. Для изменений в XML-описаниях метаданных и форм — задача быстро становится сложной:

  • Форма с десятком элементов (поля, группы, кнопки с обработчиками, таблицы, динамические списки с колонками, переключатели, картинки, декорации) — сотни связанных XML-узлов с UUID, привязками к реквизитам, командам, ресурсам. Сгенерировать без ошибок — задача не из лёгких.
  • Макет печатной формы (табличный документ) — сетка ячеек с оформлением, объединениями, ширинами колонок, параметрами, именованными областями. Через текстовое редактирование XML это практически непосильно.
  • Схема компоновки данных — наборы данных, поля, ресурсы с агрегацией, группировки, отборы (20+ типов сравнения), условное оформление, варианты. Десятки взаимосвязанных XML-фрагментов.
  • Расширение конфигурации — заимствованные объекты с правильными ссылками на базовую конфигурацию.

vibecoding1c в эту зону осознанно не идёт — их семь серверов про поиск и контекст, а не про модификацию структурных XML. Изменения метаданных и форм остаются на стороне Cursor и AI.

1C:RSV эту проблему обходит архитектурно: AI не видит XML вообще. Он вызывает операцию add_form_field, set_cell_text, add_dataset — а EDT сам собирает корректную EMF-модель и сериализует её, ровно так же, как при клике мышкой в IDE.

Про базу из 2500+ шаблонов

Это одно из самых заметных преимуществ vibecoding1c, и оно действительно ценное — особенно для типовых конфигураций. Шаблоны накапливают коллективный опыт: если задача похожа на то, с чем уже сталкивались, AI быстро находит готовое решение. Плюс есть «долговременная память» — пользователь сам пополняет базу своими паттернами.

Подход 1C:RSV принципиально другой: источник знаний — сама 1С:EDT и сам проект пользователя.

  • Код пишется в стиле проекта. Через search_in_code, read_method_source, get_method_call_hierarchy AI видит реальные паттерны конкретно вашего кода — как у вас принято выводить сообщения, формировать движения, оформлять запросы.
  • Справка платформы — встроенная, нативная. Тысячи топиков по API, языку запросов, директивам компиляции, ~130 функциям СКД — те же самые, что открываются в EDT по F1.
  • Справка по объекту — кнопка «?» в EDT. HTML-документация, заложенная разработчиком конфигурации в объект. AI получает её через get_object_help.

Какой подход лучше — зависит от ситуации. Для типовой ERP/УТ/БП библиотека шаблонов хорошо помогает. Для самописной или отраслевой конфигурации полезнее «изучить ваш код напрямую».

И ещё нюанс: чужая база шаблонов и RAG-индексы заведомо собраны под определённые версии платформы и БСП. Когда у вас самописная конфигурация, отраслевое решение или сильно кастомизированная типовая — значительная часть знания этих серверов перестаёт быть применимой: индексы не знают про ваши объекты и ваши паттерны.

А как же RAG по справке БСП?

Сервер с RAG-индексом по справке БСП — заметный пункт в линейке vibecoding1c. Полезный для типовых конфигураций. Но стоит понимать, что именно он индексирует и каков альтернативный путь.

Что такое БСП с точки зрения 1С:EDT: это набор общих модулей, встроенных в конкретную конфигурацию (ОбщегоНазначения, СтроковыеФункции, РаботаСФайлами). Их «справка» — это:

  • документирующие комментарии перед каждой функцией (параметры, возвращаемое значение, пример), которые написали разработчики БСП;
  • сам код функций — самая полная и точная документация, которая бывает.

Вся эта информация лежит внутри проекта и индексируется EDT штатно. AI получает её через те же инструменты, что и любой другой код:

  • list_modules — видит все CommonModule.*, включая модули БСП.
  • get_module_structure — карта функций модуля БСП с сигнатурами.
  • read_method_source — читает нужную функцию вместе с её документирующим комментарием.
  • search_in_code — полнотекстовый поиск по BSL, в том числе по реальным вызовам функций БСП в вашем коде.
  • find_object_references, get_method_call_hierarchy — все вызовы функции, кто её зовёт, кого зовёт она.
  • Найденные паттерны можно сохранить в .ai/project-knowledge.md — и в следующих сессиях AI начнёт сразу с них.

Ключевое отличие подходов: vibecoding1c даёт «семантический поиск по описаниям функций БСП из заранее собранного индекса» — быстро и удобно для типовой БСП. 1C:RSV даёт AI сами функции БСП в вашем проекте — с кодом, комментариями, вызовами, контекстом. Со всеми патчами, форками, локальными правками, если они есть.

Любая конфигурация, а не только типовая

Подход vibecoding1c сильнее всего раскрывается на типовых конфигурациях. Заранее собранные шаблоны, RAG-индекс по справке БСП и платформе — всё это работает тем лучше, чем ближе ваша конфигурация к стандартной ERP/УТ/БП. Это сильная сторона на типовом стеке.

Подход 1C:RSV принципиально универсален. Источник знаний — сама конфигурация, какой бы она ни была. AI работает с проектом через нативные индексы и графы 1С:EDT — те же самые, на которые опираются Go to Definition, Find References, Call Hierarchy, поиск по коду, валидатор и парсер Xtext. То есть AI видит конфигурацию глазами профессионального разработчика 1С, который только что открыл проект в EDT — но в сотни и даже тысячи раз быстрее.

Из этого следуют конкретные практические следствия:

  • Самописная конфигурация с нуля. AI открывает проект и через list_metadata_objects, ai_context, get_module_structure, read_method_source строит понимание архитектуры за минуты.
  • Отраслевое решение (медицина, строительство, ВЭД, маркетплейсы, производство, логистика). AI разбирается в специфических объектах вашей отрасли по документирующим комментариям, синонимам, подсистемам и реальным вызовам в коде.
  • Сильно кастомизированная типовая. AI работает с вашим текущим кодом, а не с тем, что было в коробке. Заранее собранный индекс этих изменений не видит.
  • Редкая или старая БСП. AI видит ровно тот код БСП, который реально лежит в вашей конфигурации.
  • Расширения с нестандартными объектами. Заимствование, доработка форм, добавление реквизитов — всё через нативные API EDT.

Грубо говоря: vibecoding1c сильнее всего работает там, где задача типовая. 1C:RSV даёт AI возможность изучить любую конкретную конфигурацию на лету, поэтому одинаково хорошо работает и на типовой, и на самописной, и на отраслевой.

Когда что выбирать

vibecoding1c

Подойдёт, если:

  • вы работаете в Конфигураторе (а не в EDT) — vibecoding1c это поддерживает, 1C:RSV нет;
  • ключевая задача AI — поиск, подсказки, ревью существующего кода, без активной модификации метаданных;
  • вам комфортна экосистема Cursor + Docker, и вы готовы поднимать семь контейнеров;
  • нужны большие RAG-индексы по справке БСП и платформы и база готовых шаблонов кода;
  • готовы дополнительно подключить 1С:Напарник для качественного ревью.
1C:RSV

Подойдёт, если:

  • вы работаете в 1С:EDT;
  • ваша конфигурация — любая: типовая, нетиповая, самописная, отраслевая, кастомизированная, на любой версии БСП;
  • вы аналитик, и нужно изучать конфигурацию: понимать, как работает механизм, что изменилось в обновлении, какие связи между объектами;
  • вы разработчик или архитектор, и нужно, чтобы AI сам создавал объекты метаданных, формы, макеты, отчёты СКД, расширения и писал BSL-код;
  • нужен полноценный отладчик 1С под управлением AI;
  • важна гарантия, что ничего не сломается — XML конфигурации останется консистентным, BSL-код будет написан органично паттернам вашего проекта, валидация выполнится автоматически после каждой записи;
  • нужен профессиональный инструмент с предсказуемым поведением, без Docker и векторных БД.

Заключение

Команда vibecoding1c сделала большое дело: показала российскому 1С-сообществу, что AI может реально помогать в разработке, а не быть игрушкой. Их продукт — серьёзный шаг вперёд для рынка, и для своих задач он отлично работает.

1C:RSV MCP Server даёт AI полный рабочий цикл аналитика, разработчика и архитектора 1С:EDT через нативные API платформы: от изучения любой конфигурации (типовой, отраслевой, самописной, на любой версии БСП) до создания справочников, форм с любыми элементами, печатных макетов, схем компоновки, расширений, написания и отладки BSL-кода. Архитектурно это сегодня единственное на рынке решение такого рода.

Честно говоря, при работе в 1С:EDT функционал vibecoding1c полностью покрывается возможностями 1C:RSV — и значительно расширяется за счёт прямого доступа к API EDT. vibecoding1c остаётся релевантным выбором главным образом для тех, кто работает в Конфигураторе (1С:EDT не использует), либо как способ начать пробовать AI-разработку на типовых конфигурациях через Cursor.
